01 — Architecture

Three layers, one goal

The platform is built as a monorepo with three independent layers. Each layer can be used on its own, or they work together as a full stack.

Layer 1 — Library

Python Package

Import directly in any Python script. Pass a dict, get a PDF. Works with pyRevit, Jupyter, CI/CD, or any Python environment.

Python 3.10+
Layer 2 — API Server

REST API

FastAPI server that accepts JSON and returns PDF files. Handles authentication, validation, template management, and multi-tenant brand resolution.

FastAPI + Docker
Layer 3 — Web Editor

Browser UI

Visual report editor built with React. Drag-and-drop blocks, live PDF preview, template scaffolding, undo/redo. Produces JSON — the API handles the rest.

React + TypeScript

05 — API Examples

Generate a PDF in three calls

Authenticate, get a scaffold, fill in your data, generate. Here are working examples.

Step 1 — Authenticate

cURL 
# Login and get a JWT token
curl -X POST https://report.example.com/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username": "admin", "password": "secret"}'

# Response:
# {"user": {"id": "...", "role": "admin"}, "token": "eyJhbG..."}

Step 2 — Get a template scaffold

cURL 
# Get a pre-filled starting point for a "structural" report
curl https://report.example.com/api/templates/structural/scaffold \
  -H "Authorization: Bearer eyJhbG..."

# Response: a complete JSON object you can fill in
# {"template": "structural", "project": "", "sections": [...], ...}

Step 3 — Generate PDF

cURL 
# Send your report JSON, receive a PDF
curl -X POST https://report.example.com/api/generate \
  -H "Authorization: Bearer eyJhbG..." \
  -H "Content-Type: application/json" \
  -d @my_report.json \
  --output report.pdf
JavaScript 
const res = await fetch("/api/generate", {
  method: "POST",
  headers: {
    "Content-Type": "application/json",
    "Authorization": `Bearer ${token}`,
  },
  body: JSON.stringify(reportData),
});
const pdf = await res.blob();
// Save or display the PDF
Three render engines: /api/generate uses the ReportLab engine (general purpose). /api/generate/v2 uses PyMuPDF for pixel-perfect stationery overlay. /api/generate/template uses the YAML template engine for tenant-specific layouts. The web editor automatically routes to the correct engine based on the selected template.

06 — Quick Start

From code to PDF in seconds

Install the package, create a Report, add content, call build(). That's it.

Shell 
# Install the package
pip install -e ".[dev]"

# Or install from the repo
pip install git+https://github.com/OpenAEC-Foundation/openaec-reports.git

Option A — Build from code

Python 
from openaec_reports import Report, A4

# Create report with metadata
report = Report(
    format=A4,
    project="Kijkduin Reddingspost",
    project_number="2026-031",
    client="Gemeente Den Haag",
    brand="default",
)

# Add pages
report.add_cover(subtitle="Constructieve berekening")
report.add_colofon()

# Add content
report.add_section(title="Staalligger L1", level=1)
report.add_calculation(
    title="Veldmoment",
    formula="M_Ed = q × l² / 8",
    substitution="M_Ed = 8.7 × 6.0² / 8",
    result="39.1",
    unit="kNm",
)
report.add_check(
    description="Unity check buiging",
    required="UC ≤ 1.0",
    calculated="M_Ed / M_Rd = 39.1 / 100.9",
    unity_check=0.39,
)

# Generate PDF
report.build("output/rapport.pdf")

Option B — Build from JSON

Python 
from openaec_reports import Report

# Load from a JSON file
report = Report.from_json("schemas/example_structural.json")
report.build("output/rapport.pdf")

# Or from a Python dict
data = {
    "template": "structural",
    "project": "My Project",
    "sections": [{
        "title": "Chapter 1",
        "content": [
            {"type": "paragraph", "text": "Hello world."}
        ]
    }]
}
report = Report.from_dict(data)
report.build("output/rapport.pdf")

Option C — CLI

Shell 
# Generate from JSON file
bm-report generate --data my_report.json --output rapport.pdf

# Start the API server
bm-report serve --port 8000 --reload

# List available templates
bm-report templates --list

# Validate a JSON file
bm-report validate --data my_report.json

The create_block() factory creates any block from a dict — this is what from_dict() and the API use internally.

from openaec_reports.core.block_registry import (
    create_block,

# Create a block from a dict
block = create_block({
    "type": "calculation",
    "title": "Moment",
    "result": "50",
    "unit": "kNm",
})

# Resolve image: path, URL, or base64
path = resolve_image_source(
    "photo.png",
    base_dir=Path("data/"),
)

JSON Schema

The contract between all three layers. Understand it, and you understand the system.

08 — Schema Structure

What a report JSON looks like

The file schemas/report.schema.json defines every field, type, and constraint. Here is the top-level structure — required fields in amber, optional in gray.

template string — which template to use project string — project name project_number string client string author string date ISO 8601 — e.g. "2026-03-01" version string — default "1.0" status enum — CONCEPT | DEFINITIEF | REVISIE format enum — A4 | A3 orientation enum — portrait | landscape cover object — cover page config colofon object — document info page toc object — table of contents sections array — main content backcover object metadata object — free-form custom data

Minimal valid report

JSON 
{
  "template": "structural",
  "project": "My first report",
  "sections": [
    {
      "title": "Introduction",
      "content": [
        {
          "type": "paragraph",
          "text": "Hello, world."
        }
      ]
    }
  ]
}
Only template and project are required. Everything else has sensible defaults. Start minimal and add fields as needed.

10 — Tenants

How tenant isolation works

Each tenant gets a directory with their own brand.yaml, templates, stationery, logos, and fonts. The engine is generic. The look and feel is per tenant.

tenants/ ├── acme_engineering/ │ ├── brand.yaml # colors, fonts, logos │ ├── templates/ │ │ ├── structural.yaml │ │ └── inspection.yaml │ ├── stationery/ │ │ ├── cover.pdf # page backgrounds │ │ ├── content.pdf │ │ └── colofon.pdf │ ├── logos/ │ │ └── logo.svg │ └── fonts/ │ ├── Brand-Bold.ttf │ └── Brand-Regular.ttf │ └── other_company/ ├── brand.yaml └── ...

brand.yaml example

YAML 
name: "ACME Engineering"
slug: "acme_engineering"

colors:
  primary: "#1E3A5F"
  secondary: "#4A90D9"
  text: "#2C3E50"

fonts:
  heading: "Brand-Bold"
  body: "Brand-Regular"

logos:
  main: "logos/logo.svg"

stationery:
  standard:
    source: "stationery/content.pdf"
  cover:
    source: "stationery/cover.pdf"

Resolution chain

When the system needs an asset (template, font, stationery), it checks in this order:

Tenant dir

tenants/acme/

→

Package assets

openaec_reports/assets/

→

Defaults

Helvetica, generic layout

Zero-config start: Without any tenant directory, the system generates valid PDFs using built-in defaults. Adding a tenant is additive — you only override what you want to customize.
11 — Security & Deployment

Production-ready from day one

JWT auth, role-based access, API keys, Docker deployment — all built in.

JWT Authentication

Login with username + password. Tokens stored as httpOnly cookies. Configurable secret via environment variable.

Role-Based Access

admin = full access. user = report editing only. Tenant scoping for multi-org deployments.

API Key Auth

Machine-to-machine authentication for scripts, CI/CD, and external integrations. Create via admin panel, set expiry dates.

Docker Deployment

Multi-stage Dockerfile: Node.js builds the frontend, Python serves everything. One container, one port, health check included.
